Campaign Name ID Format
[Placement]-[BU]-[CampaignType]-[Ads]-[Sequence]-[Audience]-[StartDate]-[EndDate]-[Year]-[StartTime]-[EndTime]_slot[N]
Example: IA-HPB-RTLNF-CF-NON-1542-All User-10 Apr-12 Apr-2026-00:00-23:59_slot 1
Business Unit Codes
AST = Astro Level (NEW)   AC = Astro Cafe   AG = Astro Goods   AB = Astro Basics   FF = Fresh/Frozen
RTLF = Retail Food   RTLNF = Retail Non Food   NB = New Buyer   DOR = Dormant
Schema & Documentation
Database schema, slot governance, automation, component structure
Operational Architecture & Scalability
1. Slot governance engine: Slot assignment is computed server-side from campaign type, date and existing occupancy — never trust client-supplied slot numbers.
2. Date-range indexing: Index campaigns on (placement, start_date, end_date) so day-level slot lookups stay fast.
3. Conflict checks: Validate slot + time overlap on every save; reject D-Day, warn on D-1.
4. Calendar queries: Month view should query only the visible date range, not the full campaign table.
5. Pagination: Campaign list pages 50 rows with virtual scroll for large volumes.
6. Partitioning: Partition the campaigns table by year for historical query performance.
7. Banner assets: Store uploaded banners in object storage (S3/GCS); keep only the URL reference in the campaign record.
